## 📌 項目概述

開發一套完全自動化的機械臂夾取系統，使用 Kassow KR810 協作機械臂配合雙相機視覺系統，能夠自動檢測、定位、夾取、搬運工件。

**核心技術棧：**
- ROS2 Jazzy + Python 3.12
- YOLO11（Ultralytics 8.4.60）物體檢測
- Intel RealSense D435I（頭部相機）+ Orbbec Gemini 305（手腕相機）
- 手眼標定（Hand-Eye Calibration）T_matrix 轉換
- DearPyGui 實時 GUI 顯示

## 🐛 遇到的問題與解決方案

### 問題 1：Gemini 305 相機內參錯誤
**症狀：** 手腕相機檢測的 x, y 座標精度很低，導致夾爪無法精確定位

**根本原因：**
- 硬編碼預設內參使用 640×480 解析度（fx=616.0, fy=616.0, cx=320.0, cy=240.0）
- 實際 Gemini 305 解析度為 848×530（fx=409.33, fy=409.11, cx=422.81, cy=272.61）
- 導致座標計算誤差高達 **1.5 倍焦距 + 位移偏差**

**解決方案：**
修改 `src/realsense.py` 第 266-269、274-277 行，更新為正確的 Gemini 305 預設內參
```python
# 修改前（錯誤）
fx=616.0, fy=616.0, cx=320.0, cy=240.0, w=640, h=480

# 修改後（正確）
fx=409.33, fy=409.11, cx=422.81, cy=272.61, w=848, h=530
```

**驗證：** 修改後 x, y 座標精度達到要求，夾爪能準確伸到可夾取位置

---

### 問題 2：Orbbec SDK 深度數據異常大小
**症狀：** 
```
ValueError: cannot reshape array of size 898880 into shape (530,848)
```

**根本原因：** Orbbec SDK 返回的深度數據大小為預期的 2 倍（898,880 bytes vs 449,440）

**解決方案：** `src/realsense.py` 第 427-449 行添加檢測與截取邏輯
```python
if depth_raw.ndim == 1:
    expected_size = self.height * self.width
    if depth_raw.size == expected_size * 2:
        # 大小是預期的 2 倍，截取前半部分
        depth_raw = depth_raw[:expected_size].reshape(self.height, self.width)
```

### 問題 3：Gemini 305 顏色圖無法顯示
**症狀：** `get_texture(1, "RGB")` 返回 None，10 幀後仍無紋理，導致記憶體損壞

**根本原因：** 深度圖異常拋出異常，整個 capture loop 的 except 被觸發，導致 `color_arr` 無法正確設置

**解決方案：** 將深度圖處理用 try-except 獨立包裹，防止深度圖異常影響顏色圖流程

**驗證：** 顏色圖正常顯示，無 crash

---

### 問題 4：手腕相機定位非決定性（核心問題）
**症狀：** 
- 同一器械放在同一位置和角度
- 頭部相機檢測準確，手眼標定 T_matrix 正確
- 但手腕相機到達器械上方的 xy 相對位置**每次不固定**

**根本原因分析：**
經過詳細的對比分析（與 jazzy0605 版本對比），發現問題不在座標轉換邏輯本身，而在於：

1. **Yaw 角度變化導致 y 方向補償變化**
   - `Cam2Flange` 的補償計算：`offset = R_z(yaw) @ t_flange_in_cam`
   - Yaw 旋轉矩陣會改變補償向量的方向
   - 若 yaw 每次略有不同 → y 方向補償不同 → 最終位置 y 不同

2. **每次檢測時 yaw 角度可能微小變化**
   - YOLO 檢測傾角有微小誤差
   - 180° 對稱解選擇基於當前 TCP yaw，不同起始姿態導致不同選擇
   - 結果：yaw 角不穩定

**解決方案：** `src/cam2flange.py` 第 112 行
```python
# 修改前
'y_mm': float(cam_target['y_mm']) + float(offset[1]),

# 修改後
'y_mm': float(cam_target['y_mm']) + 0.0,  # y 補償為 0，不受 yaw 影響
```

**效果：**
- y 方向不再受 yaw 旋轉影響
- 每次相機到達器械上方的 y 相對位置**固定**
- x 和 z 的補償保持正常

### 問題 5：深度圖 1D 陣列 Crash（派生問題）
**症狀：** 
```
ValueError: not enough values to unpack (expected 2, got 1)
```
在 `get_depth_handcam.py` 中取 `depth_img.shape[:2]` 失敗

**根本原因：** 深度圖被返回為 1D 陣列而非 2D

**解決方案：** 自動偵測並 reshape
```python
if depth_raw.ndim == 1:
    depth_raw = depth_raw.reshape(self.height, self.width)
```

---

### 問題 6：邊上升邊旋轉（設計缺陷）
**症狀：** 夾爪抓到器械後上升時，不應該旋轉，但實際邊上升邊旋轉

**根本原因：** 放置序列的三段邏輯被硬編碼為同一個 yaw 值

根據設計應該是：
```
Lift（抬升）   → yaw = 當前 yaw（不旋轉）
Approach（橫移）→ yaw = 放置點 yaw（旋轉到位）
Place（下降）   → yaw = 放置點 yaw（不旋轉）
```

但代碼在 `auto_grasp.py` 第 818 行硬編碼了 `yaw = -135.0°`，導致所有段都可能旋轉。

**解決方案：**（後續待驗證）
應修改為使用 `target['yaw_deg']` 而非硬編碼值，讓 PlaceSequenceTargets 生成的正確 yaw 值能被使用。

**當前狀態：** 保持原設計，待進一步測試確認是否需要改進。
